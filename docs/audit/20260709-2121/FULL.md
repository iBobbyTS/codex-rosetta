## Finding F-01 — Admin configuration API returns the plaintext Admin password

- **Status:** Reviewed
- **Severity:** Must Fix
- **Scope:** `src/codex_rosetta/gateway/admin/routes/config.py:42-64`, `get_config()` at `:445-495`; default literal secret generation in `src/codex_rosetta/gateway/cli.py:_secure_server_template`.
- **Focus:** Authentication secret exposure, least privilege, browser/admin trust boundary.
- **Finding:** `_mask_server_config()` masks gateway API keys and Tavily keys but leaves `server.admin_password` intact. `get_config()` returns that object even when `credential_visible` is false. The secure default generated by `init` is a literal password, so the default deployment exposes the durable login credential to every authenticated Admin browser response and stores it in the SPA's `configData` object.
- **Trigger / failure path:** Authenticate once, then `GET /admin/api/config` with `X-Admin-Token`. The response includes `server.admin_password` verbatim.
- **Impact:** A compromised Admin bearer token, browser extension, or future Admin-page XSS gains the reusable password in addition to the process-scoped token. This defeats credential minimization and conflicts with the secure-default claim that credential reveal is disabled.
- **Evidence:** An isolated app configured with `admin_password="durable-admin-secret"` and `credential_visible=false` returned `200 False durable-admin-secret gate***cret`; the gateway key was masked while the Admin password was not.
- **Recommended priority:** Before merge/release of the security hardening.
- **Suggested fix direction:** Remove `admin_password` from the Admin config response entirely (the UI does not consume it), or return a non-secret configured/masked sentinel. Add a route-level regression asserting the literal password never appears for either credential-visibility setting or env-placeholder configs.

## Finding F-02 — Access-key creation can persist an invalid, startup-blocking config

- **Status:** Reviewed
- **Severity:** Must Fix
- **Scope:** `src/codex_rosetta/gateway/admin/routes/keys.py:create_api_key()` (`:44-102`); `src/codex_rosetta/gateway/config.py:GatewayConfig._validate()` (`:396-433`); `gateway/admin/routes/_shared.py:_reload_gateway_config()` (`:45-58`).
- **Focus:** Config transaction semantics, validation boundary, recoverability.
- **Finding:** `create_api_key()` accepts an arbitrary manual `body.key`, appends it, and atomically writes the file before constructing `GatewayConfig`. Duplicate/non-string/unresolved-placeholder keys are therefore persisted first; validation fails only during hot reload, leaving the live app on the old config while the on-disk config is invalid.
- **Trigger / failure path:** Create a second key with the same manual key value as an existing entry. The route returns HTTP 500 with `{"saved": true, "reloaded": false}`, and the next process start raises `ValueError: config: duplicate server.api_keys[].key`.
- **Impact:** A normal Admin API request can make the persisted gateway configuration unbootable. The `.bak` file makes manual recovery possible, but the service does not roll back automatically and the API does not reject the invalid request before commit.
- **Evidence:** Isolated route reproduction returned `500 {"error": "Config saved but reload failed: config: duplicate server.api_keys[].key", "saved": true, "reloaded": false}`; re-parsing the saved file raised the same `ValueError`.
- **Recommended priority:** Before merge/release.
- **Suggested fix direction:** Validate the candidate key and whole candidate config before `write_config()`. Prefer a shared mutate → substitute/validate candidate → CAS write → swap-live-state transaction used by every Admin config mutation, with 400/409 for caller errors and no disk mutation. Add duplicate, non-string, unresolved-placeholder, and simulated reload-failure tests.

## Finding F-03 — Secret redaction is not enforced at every persisted-diagnostic boundary

- **Status:** Reviewed
- **Severity:** Must Fix
- **Scope:** `src/codex_rosetta/observability/persistence.py:insert_log_entries()` (`:303-342`), `update_entry_profile()` (`:904-933`); `observability/error_dump.py:_dump_error_impl()` (`:205-255`); producers in `gateway/app.py:330-384` and `gateway/proxy.py:_finalize_stream_profile` / `_raw_stream_event_generator`.
- **Focus:** API-token leakage into SQLite diagnostics.
- **Finding:** The new `SecretRedactor` is applied to error dump bodies/response text and stream JSONL events, but not to request-log `error_detail`, initial `profile`, later `profile_update` (including `stream_error`), or error-dump `upstream_url`.
- **Trigger / failure path:** An upstream error/exception contains `Bearer ...` or a configured provider/gateway token, or a provider base URL embeds credentials/query tokens. The raw value is committed to `gateway.db` and exposed by Admin log/error-dump APIs.
- **Impact:** Credential material survives in the exact persistence surface the security hardening is intended to protect. This violates the audit profile's explicit “credentials are not logged accidentally” target and the user-facing diagnostic-redaction statement.
- **Evidence:** A `PersistenceManager(token_values={"provider-secret"})` stored `error_detail='Bearer bearer-secret provider-secret'` and `profile.stream_error='provider-secret'` unchanged; `update_entry_profile()` also stored `Bearer second-secret provider-secret` unchanged. `dump_error(... upstream_url='https://user:provider-secret@example.test/v1?key=provider-secret')` returned the same raw URL from SQLite.
- **Recommended priority:** Before merge/release.
- **Suggested fix direction:** Make redaction a single invariant of persistence write methods, not a caller convention. Redact a copy of each log entry/profile update and every text metadata field in `insert_error_dump()`; add regression tests that query SQLite after insert/update and verify the raw tokens do not occur anywhere.

## Finding F-04 — Configured cross-origin Admin access cannot pass browser preflight

- **Status:** Reviewed
- **Severity:** Should Plan
- **Scope:** `src/codex_rosetta/gateway/auth.py:check_admin_auth()` (`:72-99`); `gateway/app.py` CORS/preflight setup (`:630-682`).
- **Focus:** Optional Admin CORS contract, auth/CORS middleware ordering.
- **Finding:** `server.admin_cors_origins` is documented and `_apply_cors()` can build a valid allowed-origin response, but the before-request auth hook rejects unauthenticated `OPTIONS /admin/api/*` before the catch-all preflight route runs. Browsers do not include the actual `X-Admin-Token` value in a preflight, so authenticated cross-origin Admin requests are impossible.
- **Trigger / failure path:** Preflight `OPTIONS /admin/api/config` with an allowed `Origin`, requested method, and requested `x-admin-token` header.
- **Impact:** An explicit configuration feature silently does not work; operators may misdiagnose it as proxy/browser failure.
- **Evidence:** Isolated dispatch with `admin_cors_origins=["https://admin.example"]` returned HTTP 401 and no CORS headers instead of the intended 204 response.
- **Recommended priority:** Fix before advertising or depending on cross-origin Admin hosting.
- **Suggested fix direction:** Let `OPTIONS` reach the CORS handler without Admin credentials, while still reflecting only configured origins. Add allowed/disallowed-origin preflight tests and actual authenticated request tests.

## Finding F-06 — Diagnostic storage has no total-size or rotation budget

- **Status:** Reviewed
- **Severity:** Track as Debt
- **Scope:** `observability/error_dump.py:35-36`; `observability/persistence.py:658-659`; `gateway/stream_trace.py:19-30,150-181`; `docs/en/gateway-security.md:56-68` and Chinese counterpart.
- **Focus:** Availability, disk exhaustion, operability.
- **Finding:** Error dumps are capped only at 10,000 rows while each original and converted body may be up to 10 MiB before compression. Optional stream trace appends every stage/chunk to one JSONL file with per-string truncation but no file-size limit, rotation, age policy, or emergency disable based on disk state.
- **Impact:** Error-heavy or long-traced workloads can consume tens/hundreds of GB and take down the gateway/host filesystem. Error-dump limits are documented, but no operational budget or rotation mechanism exists; stream-trace growth is not documented.
- **Recommended priority:** Track with an owner now; promote before sustained remote/multi-user operation or enabling stream trace broadly.
- **Owner decision / accepted debt:** Gateway maintainers and deployment operators explicitly retain token-only redaction plus the existing 10,000-row count-only error-dump policy. No age/size pruning or broader content/password/secret redaction will be added. Revisit only before sustained remote/multi-user operation, broad long-running stream tracing, a disk-capacity incident/alert, or an operator requirement for a hard storage budget.

## Finding F-07 — Request-log client attribution trusts spoofable forwarded headers

- **Status:** Reviewed
- **Severity:** Track as Debt
- **Scope:** `src/codex_rosetta/gateway/app.py:_extract_client_ip()` (`:115-131`); `gateway/admin/routes/auth.py:_get_client_ip()` (`:42-52`); `docs/en/gateway-security.md:51-54`.
- **Focus:** Audit-log integrity and trusted-proxy boundary.
- **Finding:** Login limiting deliberately ignores forwarded headers because no trusted-proxy allowlist exists, but request logging unconditionally trusts the first `X-Forwarded-For` or `X-Real-IP` value. Any authenticated direct client can spoof the IP shown in Admin logs.
- **Impact:** This does not bypass authorization, but it makes incident/audit attribution unreliable and creates contradictory proxy semantics.
- **Suggested fix direction:** Use the direct peer by default in both paths; only honor forwarded headers when the direct peer matches an explicit trusted-proxy configuration, and retain both peer and forwarded chain when enabled.
