# Built-in Code Mode Tool Test

This suite covers the built-in tools in the maintained Codex compatibility
ledger for an OpenAI-identified provider, using `gpt-5.6-sol` as the default
model-shape reference with the local-mode catalog. It tests tool exposure, Code
Mode projection, Rosetta reconstruction, native execution, and result delivery.
It does not measure general coding, planning, visual, or agent quality.

## Fixed scope

Every run in this suite must use:

- a provider whose display name is exactly `OpenAI`;
- the model metadata generated by local mode, with `gpt-5.6-sol` used only as
  the reference shape;
- the bundled `Chat Default` Tool Profile;
- a local environment and the platform's normal Codex shell configuration;
- one isolated task and Codex home per timestamp run.

The Codex-facing model alias and actual upstream model may differ, but Gateway
Logs must prove both. Retain the catalog generated by local mode.

## Scenarios

- `01`: start a yielded Code Mode `exec` cell and resume it with the top-level
  `wait` Function.
- `02`: call projected `update_plan` twice and verify both native plan updates.
- `03`: give the model a protocol-neutral file-modification outcome. On a Chat
  route, observe whether it naturally selects all five localized `Glob`,
  `Grep`, `Read`, `Edit`, and `Write` Functions. On a direct GPT Responses
  route, observe whether GPT selects native `apply_patch`. The prompt does not
  prescribe either tool surface; file outcomes and route-specific tool
  selection are evaluated separately.
- `04`: call projected `view_image` on a deterministic local PNG fixture and
  return its image output through Code Mode.
- `05`: call `get_goal`, `create_goal`, and `update_goal` in one fresh-thread
  lifecycle.
- `06`: call projected `view_image` on a deterministic four-quadrant PNG and
  verify that the default multimodal third-party model `mimo-v2.5` identifies
  all four colors and positions. Gateway Logs must prove the actual upstream.

Run every task separately. State created by the plan, file-editing, or Goal
scenario must not leak into another run.

## `request_user_input` limitation

There is intentionally no `request_user_input` task for the current
`codex exec` runner. The exact version-bound rejection observation is retained
in the
[central compatibility evidence](../../../docs/dev/version-compatibility/evidence/real_agent_tool_test_results.md).
The upstream Codex test uses an app-server test client: it receives
`ToolRequestUserInput`, captures the JSON-RPC request id, and explicitly sends
an answer response before waiting for turn completion.

An honest real-agent test therefore requires a separate app-server JSON-RPC
runner with deterministic response injection. Do not substitute
`auto_resolution_ms`: the core session only forwards that value to the client,
while `codex exec` still rejects the request. Until such a runner is added,
record this tool as `runner_not_supported`, not `not_exposed`, `not_called`, or
a model failure.

## Required configuration

Use the isolated local-mode Provider ID `codex_rosetta` generated by the
current gateway and require its display name to be exactly `OpenAI`. Enable
the stable Goal feature explicitly for task `05` so the test does not depend on
future default changes. Do not enable Plan collaboration mode for task `02`:
Codex defines `update_plan` as a TODO/checklist tool and rejects it in Plan
mode.

Follow `EVALUATION.md` and write `artifacts/evaluation.json` for every run.
